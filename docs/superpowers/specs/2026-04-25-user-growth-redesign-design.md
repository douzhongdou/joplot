# 用户增长导向改版设计

日期：2026-04-25

## 目标

围绕 `joplot` 做一轮以“用户增长”为导向的产品改版，用尽量少的交互成本，把用户从“打开页面”推进到“理解价值、立即出图、愿意传播”。

这轮改版不是做一批分散的小优化，而是统一围绕下面这条增长路径设计：

```text
进入页面 -> 3 秒理解产品价值 -> 立即试出结果 -> 复制/下载成果 -> 自然传播 -> 再次回访
```

本轮设计聚焦纯前端可落地路径，不引入后端、账号、云端保存或高运维依赖。

## 产品原则

### 1. 默认优于配置

用户不应该在第一次使用时面对太多设置项。优先提供“默认就好用”的结果，而不是把决定权全部推给用户。

具体体现：

- 图片导出默认白底，不提供透明背景开关
- 首次上传后自动生成可用图表
- 默认文案优先解释“你能立刻得到什么”
- 分享链路优先一键完成，而不是给一串导出参数

### 2. 先让用户爽，再让用户学

首次体验的重点不是完整掌握工作台，而是：

- 上传很快
- 出图很快
- 图看起来就能用
- 复制/下载立刻可发给别人

进阶配置仍然保留，但不应阻塞首个结果出现。

### 3. 分享结果比学习产品更重要

增长不是靠用户理解全部能力，而是靠用户把成果带出去。

因此本轮所有优先级都围绕三个问题：

- 用户能不能更快拿到一张可用图
- 用户能不能更自然地复制或下载出去
- 用户分享后的接收方能不能理解 `joplot` 是什么

### 4. 增长优化不能破坏轻量感

`joplot` 当前优势是快、轻、本地优先、零注册、低摩擦。本轮改版应强化这些优势，而不是把产品做成一个需要教学的复杂 BI 工具。

## 当前现状

### 1. 已有能力

当前产品已经具备增长改版的基础能力：

- 支持 CSV / Excel 上传，包含按钮上传和整页拖拽上传：[src/lib/upload.ts](</D:/HQL/code/tool/joplot/src/lib/upload.ts:19>)、[src/App.tsx](</D:/HQL/code/tool/joplot/src/App.tsx:317>)
- 首次导入后会自动创建默认图卡：[src/App.tsx](</D:/HQL/code/tool/joplot/src/App.tsx:234>)
- 已支持复制图片、下载图片，以及剪贴板失败时回退下载：[src/components/PlotCanvas.tsx](</D:/HQL/code/tool/joplot/src/components/PlotCanvas.tsx:63>)
- 已支持中英日多语言与静态 SEO 入口
- 已有本地持久化和多图卡工作台

这意味着本轮增长改版不需要从零开始，而是要把现有能力重新组织成更顺滑的首用路径。

### 2. 当前阻碍增长的关键问题

#### 首页仍然偏“导入入口”，不够像“可转化的产品首页”

当前无数据状态只展示导入标题、简短描述和上传按钮：[src/App.tsx](</D:/HQL/code/tool/joplot/src/App.tsx:585>)。

问题在于：

- 用户不能立刻看到最终成果长什么样
- 用户不能快速理解“为什么值得试”
- 用户没有样例数据入口，手头没文件时会直接流失

#### 当前导航与头部更偏工作台，不偏转化

当前顶部结构主要围绕视图切换、加组件、筛选、重置数据：[src/components/AppNavbar.tsx](</D:/HQL/code/tool/joplot/src/components/AppNavbar.tsx:19>)、[src/components/WorkbenchHeader.tsx](</D:/HQL/code/tool/joplot/src/components/WorkbenchHeader.tsx:80>)。

这对已有数据的工作流是合理的，但对首次访问用户来说，不够聚焦“马上试一下”。

#### 图片导出还没有达到“拿来即分享”的标准

当前复制和下载图片走同一条 Plotly 导出路径：[src/components/PlotCanvas.tsx](</D:/HQL/code/tool/joplot/src/components/PlotCanvas.tsx:34>)。

但页面渲染时图表外层背景仍是透明：[src/components/PlotCanvas.tsx](</D:/HQL/code/tool/joplot/src/components/PlotCanvas.tsx:111>)，导致导出的 PNG 在聊天工具、文档和 PPT 中不够稳定。

本轮需要明确统一策略：默认全部白底，无透明背景设置。

#### 当前“复制成功”只完成动作，没有放大传播

复制链路已经有 toast 能力，但还没有把“复制成功”进一步转成品牌传播或二次动作引导。也没有围绕分享行为组织更明确的命名、文件名、成功反馈和埋点。

## 本轮范围

本轮作为“用户增长导向改版”的第一阶段，聚焦 P0 + 一部分对增长有直接帮助的结构性能力。

### 本轮纳入

1. 首页首屏与无数据状态重构
2. Sample Data 入口
3. 图片复制与下载体验升级
4. 分享成功反馈与轻传播提示
5. 增长事件埋点
6. 页面速度与首屏路径优化
7. 面向 SEO Landing Pages 的结构预埋

### 本轮不纳入

- 后端与账号系统
- 云端存储
- 权限、协作、分享链接系统
- 大规模模板市场
- PWA 与离线完整能力
- 本地历史记录面板
- 批量导出

## 总体方案

本轮改版采用“双态首页”的方式：

- 无数据时：产品首页态，强调价值表达、样例入口、快速开始
- 有数据时：工作台态，强调分析、编辑、复制、下载

这样既不牺牲现有工作台效率，也能把首次访问转化做起来。

## 一、首页与空状态重构

### 目标

让用户在 3 秒内理解：

```text
上传表格 -> 自动出图 -> 复制或下载
```

### 现状

当前无数据状态内容太少，只能看出“这是一个上传工具”，看不出“这是一个能快速把 CSV 变成可分享图表的产品”。

### 新方案

无数据状态改造成轻量首页，而不是简化版上传面板。

建议结构：

```text
Hero
- 标题：强调 CSV -> Chart 的即时价值
- 副标题：强调 No signup / 本地优先 / 上传即出图
- 主 CTA：上传文件
- 次 CTA：Try Sample Data

Proof strip
- 支持 CSV / Excel
- 本地处理
- 可复制图片
- 无需注册

Visual preview
- 一张默认样例图预览
- 或 2-3 张小型示例卡片预览

Sample data shortcuts
- Sales
- Stock
- Traffic
- Sensor
```

### 设计要求

- 首屏主按钮文案必须直接表达结果导向，不要过于抽象
- 样例入口必须在首屏可见，不要藏进帮助菜单
- 不要让首屏像“控制台”或“后台”
- 视觉上强调“拿结果”而不是“学系统”

### 实现方向

- 复用现有无数据状态位置：[src/App.tsx](</D:/HQL/code/tool/joplot/src/App.tsx:585>)
- 把当前简单上传区域替换为带样例和结果预览的首页组件
- 上传后仍进入现有工作台，不重做主数据流

## 二、Sample Data

### 目标

消除“我现在手上没有文件，先不试了”的流失点。

### 新方案

内置一组纯前端本地样例数据，用户点击后直接载入数据集并生成默认图卡。

首批建议包含：

- `Sales.csv`
- `Stock.csv`
- `Traffic.csv`
- `Sensor.csv`

### 设计原则

- 一键进入，不需要二次确认
- 点击样例后，体验应与上传真实文件尽量一致
- 样例数据名称和字段要足够直观，方便用户快速理解图表为何生成成那样

### 实现方向

- 使用 `public/` 或源码内嵌静态样例文件
- 通过现有 `parseFiles` / 数据导入管线统一进入工作台
- 首次加载样例时自动生成默认图卡，保持与真实上传一致

### 增长价值

- 提高首次体验完成率
- 让社媒、SEO 进入的冷启动用户也能立即试用
- 为未来 Landing Page 复用示例数据提供素材

## 三、图片复制与下载体验升级

### 目标

把“生成图表”真正变成“生成可分享成果”。

### 当前问题

- 图片底色不稳定
- 导出结果未完全以“拿去即用”为标准
- 复制与下载还没有围绕增长目标统一体验

### 明确产品决策

本轮直接采用最小默认策略：

- 导出图片统一为白底
- 不提供透明背景设置
- 导出结果保持无边框、极简图本体
- 复制和下载行为保持一致

### 行为定义

- 点击“复制图片”：
  - 输出白底 PNG
  - 优先写入系统剪贴板
  - 剪贴板不支持时自动回退为下载 PNG
- 点击“下载图片”：
  - 直接下载白底 PNG
- 三条路径都必须保证结果一致：
  - 复制成功
  - 复制失败后自动下载
  - 直接下载

### 文件名策略

下载文件名从泛化命名改成更适合传播与整理的命名：

```text
joplot-line-chart-2026-04-25.png
```

如果图卡标题可用，则优先考虑：

```text
joplot-sales-trend-2026-04-25.png
```

需要做安全字符清洗，避免空格、斜杠和非法字符。

### 后续但不在本轮默认开放的能力

- SVG 导出
- JPG 导出
- 清晰度等级切换

这些能力可以作为后续体验增强，但本轮默认只做“白底 PNG 拿来就能发”。

### 实现方向

- 统一收敛到 [src/components/PlotCanvas.tsx](</D:/HQL/code/tool/joplot/src/components/PlotCanvas.tsx:34>) 的导出链路处理
- 页面内图表可继续保留当前视觉风格，但导出结果必须覆盖为白底
- 如实现成本更低，也可直接统一页面与导出为白底，只要不破坏当前极简体验

## 四、导出成功后的轻传播引导

### 目标

让用户完成“复制/下载”后，顺手形成品牌记忆，而不是动作完成后立刻断掉。

### 新方案

在复制成功和下载成功后的反馈里，加入轻量传播提示，但不能太重、太打扰。

建议文案方向：

- `Copied. Ready for docs, slides, and chat.`
- `Downloaded. Made with joplot.`

中文与日文同步本地化，不做生硬营销口号。

### 约束

- 不要弹出分享弹窗
- 不要要求用户点击二次分享
- 不要在导出图片里强行加水印

目标是增强品牌记忆，不是打断任务流。

## 五、增长埋点

### 目标

让后续增长优化可以基于数据而不是感觉继续推进。

### 本轮最低事件集

- `visit_home`
- `use_sample_data`
- `upload_file`
- `parse_success`
- `parse_failed`
- `generate_chart`
- `copy_image`
- `copy_image_fallback_download`
- `download_png`
- `switch_language`
- `open_data_view`

### 指标重点

- 首屏访问 -> 上传转化率
- 首屏访问 -> 样例体验转化率
- 上传 -> 出图率
- 出图 -> 复制率
- 出图 -> 下载率
- 复制失败率

### 实现原则

- 事件命名保持稳定
- 埋点不影响核心交互性能
- 先做最少可判断集，不做重分析系统

## 六、页面速度与首屏体验优化

### 目标

保持“快”的产品印象，降低首次等待焦虑。

### 新方案

#### 1. 上传与解析反馈更明确

当前产品已有上传入口和拖拽覆盖层，但缺少“解析中、生成中”的过程反馈。

需要增加阶段性反馈：

- `Uploading...`
- `Parsing data...`
- `Generating chart...`

如果解析失败，也要给出清晰失败反馈，而不是静默失败。

#### 2. 无数据首屏尽量轻

- 减少不必要的首屏复杂组件
- 首屏视觉预览可优先使用静态或低成本内容
- 避免在无数据状态预先挂载重量级工作台结构

#### 3. 样例数据加载要非常快

样例文件要小而代表性强，避免“点了样例还要等很久”。

## 七、SEO Landing Pages 预埋

### 目标

为后续自然搜索流量做结构准备，但不在本轮一次性把所有落地页全做完。

### 本轮设计

先定义落地页体系与内容组件模型，后续按优先级逐个补页。

建议首批目标路径：

- `/csv-line-chart`
- `/csv-bar-chart`
- `/csv-scatter-plot`
- `/plot-csv-online`

### 每页内容结构

- H1
- 场景描述
- 示例图
- 3 步使用说明
- FAQ
- CTA（上传文件 / 试用样例）

### 与当前系统的关系

当前项目已经有多语言静态入口和基础 SEO 能力，这一轮不重做 SEO 基础设施，而是把“首页多语言 SEO”进一步扩展到“按搜索意图拆分的落地页体系”。

## 八、界面组织原则

### 1. 无数据时是“产品首页”

不应该直接让用户面对完整工作台结构。

### 2. 有数据时是“工作台”

一旦用户进入分析状态，交互重心切换为：

- 增删图卡
- 筛选
- 数据查看
- 复制下载

### 3. 高级能力不抢首屏

例如：

- 图表类型扩展
- 数据视图
- 复杂筛选
- 系列配置

这些能力仍然保留，但不应在无数据首屏抢走注意力。

## 十一、验收标准

### 首页与样例

- 无数据状态首屏能够在不阅读长文档的前提下解释产品价值
- 首屏必须同时提供上传入口和样例入口
- 点击样例后 1 次操作内进入可用工作台

### 导出体验

- 复制图片得到白底 PNG
- 下载图片得到白底 PNG
- 剪贴板失败回退下载时，仍得到白底 PNG
- 导出图保持无边框极简风格

### 反馈与传播

- 导出成功存在轻量反馈
- 不引入强打断式营销弹窗
- 用户可自然感知品牌来源，但不会被强迫传播

### 埋点

- 最低事件集可正常上报
- 至少可以区分上传、样例、出图、复制、下载、失败几类关键行为

---

## 十二、风险与取舍

### 风险 1：首屏做得太“营销页”，削弱产品感

应对：

- 继续保持上传入口在首屏核心位置
- 首屏内容为“快速试用”服务，而不是堆营销段落

### 风险 2：为了增长做太多设置项，反而破坏丝滑体验

应对：

- 本轮明确坚持“默认优于配置”
- 类似透明背景、导出格式、清晰度选择等能力先不开放或延后

### 风险 3：增长功能与工作台逻辑相互干扰

应对：

- 无数据与有数据两种状态分层设计
- 复用现有导入、出图、导出链路，不重建主数据流

---

## 十三、结论

这轮“用户增长导向改版”的核心，不是把 `joplot` 做得更复杂，而是把现有能力重新编排成一个更顺滑的增长漏斗：

```text
更快理解 -> 更快出图 -> 更容易分享 -> 更容易传播
```

其中第一阶段最重要的不是做很多功能，而是把以下几件事做到位：

1. 首屏让人立刻懂
2. 没有文件也能立刻试
3. 复制和下载拿到的图直接可用
4. 产品在分享之后还能留下名字

如果这四件事做好，`joplot` 才会真正具备自然增长的基础。
