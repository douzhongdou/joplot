# 图卡内聚合设计

日期：2026-04-23

## 目标

把当前“一个数据系列绑定一个文件和一个数值列”的图卡模型，升级为“每张图卡独立配置数据管线”的模型。

第一阶段聚焦图卡内聚合，不做全局聚合，也不做可复用数据视图。用户可以在不同图卡中分别设置不同的数据源、X 轴字段、时间分桶、分组方式、指标字段和聚合方式。例如：

- 折线图 A：按月份汇总销售额
- 柱状图 B：按品类统计订单数
- 折线图 C：按地区拆分每月平均客单价

每张图卡的聚合配置互不影响。全局状态只继续负责数据集、筛选条件、画布布局和当前选中图卡。

## 设计原则

- 聚合属于图卡，不属于全局工作台。
- 图卡类型只负责展示，数据准备统一走同一条数据管线。
- 支持多数据源，但系列拆分策略必须清晰，不能同时按文件和按字段拆分。
- 先把图卡内聚合跑通，数据视图标签页留给后续阶段。
- 保持现有原始数据作图能力可用，并提供兼容迁移路径。

## 当前模型问题

当前核心模型大致是：

- `CsvData` 保存文件级数据集
- `ChartCard` 保存图卡配置
- `ChartSeries` 绑定 `datasetId + yColumn`
- `ChartCard.xColumn` 作为图卡公共 X 轴

这套模型适合直接绘制原始行数据，但不适合表达：

- 按月份、季度、年份汇总
- 按字段值拆分系列
- 多文件合并后聚合
- 标准差、方差、去重计数等统计结果
- 同一画布中不同图卡使用完全不同的聚合口径

因此第一阶段需要新增“图卡数据配置”和“聚合输出”两层抽象。

## 图卡数据模式

每张图卡新增统一的数据配置 `dataConfig`。

数据模式分为两类：

- 原始数据模式：保留当前 `series -> datasetId + yColumn` 的直接绘图方式。
- 聚合数据模式：图卡先根据聚合配置生成标准系列，再交给图表渲染。

原始数据模式用于兼容现有图卡，也适合逐行散点、逐条记录趋势等场景。

聚合数据模式用于统计型图表。它不依赖全局聚合状态，每张图卡单独保存完整配置。

## 聚合配置

聚合模式下，图卡需要配置：

- 数据源：一个或多个数据集
- X 轴字段：用于分桶或分类
- X 轴字段类型：文本、数值、时间
- 时间粒度：仅当 X 轴为时间时可选
- 分组方式：按文件分组或按字段分组
- 分组字段：仅当选择按字段分组时需要
- 指标字段：参与聚合的数值字段
- 聚合方式：求和、平均值、计数等

配置属于单张图卡。两张同类型图表也可以使用完全不同的聚合配置。

## 多数据源聚合

第一阶段支持多个数据源参与聚合。

多数据源的系列拆分只允许两种方式二选一：

### 按文件分组

每个数据源独立执行同一套聚合规则，并生成一个系列。

适用场景：

- 多个 CSV 是不同批次、不同门店、不同实验组
- 用户想比较多个文件的同一指标走势

规则：

- 不能再选择分组字段
- 系列名默认使用文件名
- 每个数据源必须包含 X 轴字段和指标字段
- 缺少必要字段的数据源跳过，并在 UI 中提示文件名和原因

### 按字段分组

多个数据源先按同名字段纵向合并参与聚合，再按分组字段的字段值生成系列。

适用场景：

- 多个 CSV 是同一业务表的分片
- 用户想看 `月份 + 地区 -> 销售额求和`

规则：

- 必须选择分组字段
- 不再按文件拆分系列
- 同名分组值跨数据源合并
- 缺少 X 轴字段、指标字段或分组字段的数据源跳过，并在 UI 中提示文件名和原因

第一阶段不做字段映射。不同数据源之间必须使用同名字段才能共同参与聚合。

## 时间分桶

CSV 中的时间字段本质是字符串，因此第一阶段采用“常见格式解析 + 用户选择字段”的方式。

导入或配置时可以抽样识别常见日期格式：

- `YYYY-MM-DD`
- `YYYY/MM/DD`
- `YYYY-MM-DD HH:mm:ss`
- ISO 日期时间
- 常见中文日期格式

当用户把某个字段作为时间 X 轴使用时，开放以下时间粒度：

- 日
- 周
- 月
- 季度
- 年

解析失败的行不参与该图卡的聚合结果，并在图卡配置区提示失败行数。第一阶段不提供自定义日期格式输入。

## 聚合方式

第一阶段支持以下聚合方式：

- 求和
- 平均值
- 计数
- 最大值
- 最小值
- 中位数
- 去重计数
- 空值计数
- 标准差
- 方差

其中：

- 计数统计参与聚合的行数。
- 空值计数统计指标字段为空或无法解析为有效数值的行数。
- 去重计数基于指标字段原始值去重。
- 标准差和方差第一阶段使用样本统计还是总体统计，需要在实现时固定为一种并在 UI 文案中说明。建议优先使用总体统计，便于和图卡内完整分组结果对应。

分位数、累计求和、占比、环比、同比先不进入第一阶段。

## 聚合输出

聚合引擎输出标准化结果，而不是直接输出 Plotly 数据。

建议输出结构表达为：

- `xValue`：原始 X 桶值
- `xLabel`：展示用 X 标签
- `seriesKey`：系列唯一键
- `seriesName`：系列展示名
- `value`：聚合后的数值
- `sourceDatasetIds`：参与该结果的数据源
- `skippedRows`：当前配置下被跳过的行数信息

图表渲染层再把这个结果转换为 Plotly 需要的 `x/y/name` 数据。

这样后续新增饼图、环形图、雷达图时，可以继续复用同一份聚合输出。

## 图卡配置 UI

右侧检查面板继续保留“基础”和“显示”两个页签。

### 基础页

基础页新增数据模式选择：

- 原始数据
- 聚合数据

选择原始数据时，继续显示当前系列配置。

选择聚合数据时，显示聚合配置：

- 数据源多选
- X 轴字段
- X 轴字段类型
- 时间粒度
- 分组方式
- 分组字段
- 指标字段
- 聚合方式

当用户选择的配置不完整时，图卡展示明确空状态，不渲染错误图表。

### 显示页

显示页做一次结构清理：

- 删除当前不可用的主题色占位项
- 图例开关改为真正的 Switch 组件
- 网格线开关改为真正的 Switch 组件
- 坐标轴开关改为真正的 Switch 组件
- 新增 X 轴范围设置
- 新增 Y 轴范围设置

范围设置支持留空。留空表示交给 Plotly 自动范围。

X 轴范围按字段类型处理：

- 数值轴：最小值、最大值
- 时间轴：开始时间、结束时间
- 文本或分类轴：第一阶段允许选择或输入分类边界，底层按 Plotly 分类范围能力处理；如果当前图表类型不支持可靠范围，则 UI 给出提示并退回自动范围

Y 轴范围使用最小值、最大值。

## 筛选关系

第一阶段继续沿用工作台级筛选。

聚合前先应用当前工作台筛选条件，再对过滤后的行执行图卡内聚合。

这意味着：

- 筛选是全局观察口径
- 聚合是图卡局部计算口径
- 同一个筛选条件会影响所有图卡
- 不同图卡仍然可以拥有不同聚合配置

后续如果需要图卡级筛选，应单独设计，避免和本次聚合配置混在一起。

## 数据视图标签页

数据视图不进入第一阶段实现，但设计上为后续预留。

后续可以新增与图卡工作台并列的标签页：

- 图卡工作台：当前画布和图卡配置
- 数据视图：原始数据、字段概况、聚合结果预览、可复用加工结果

第一阶段的聚合引擎应放在可复用的业务函数中，避免未来数据视图标签页重新实现一套聚合逻辑。

## 兼容与迁移

现有持久化图卡没有 `dataConfig`。

迁移规则：

- 旧图卡默认视为原始数据模式
- 旧 `xColumn` 保留为原始数据模式的公共 X 轴
- 旧 `series` 保留为原始数据系列
- 旧 `yMin/yMax` 保留为 Y 轴范围设置
- 新增 X 轴范围字段默认为空

这样刷新旧工作台时不应丢失当前图卡。

## 测试策略

核心测试应覆盖：

- 原始数据模式兼容旧图卡
- 单数据源按普通字段聚合
- 单数据源按时间字段进行日、周、月、季度、年分桶
- 多数据源按文件分组聚合
- 多数据源按字段分组聚合
- 缺字段数据源被跳过并返回原因
- 空值、非法数值、非法时间被统计或跳过
- 求和、平均值、计数、最大值、最小值、中位数、去重计数、空值计数、标准差、方差
- 图卡级聚合配置互不影响
- X/Y 轴范围留空时保持自动范围
- X/Y 轴范围设置后传递到 Plotly layout

## 第一阶段不做

- 可保存、可复用的数据视图
- 字段映射
- 自定义日期格式输入
- 分位数
- 累计求和
- 占比
- 环比和同比
- 图卡级筛选
- 新文件格式导入
- 新图表类型

## 验收标准

- 用户可以在同一个画布中创建两张同类型图卡，并为它们配置不同聚合口径。
- 用户可以选择多个数据源，并按文件分组生成多个系列。
- 用户可以选择多个数据源，并按字段分组生成多个系列。
- 用户可以按日、周、月、季度、年汇总时间字段。
- 用户可以使用第一阶段定义的全部聚合方式。
- 缺少必要字段或解析失败的数据不会导致整张图崩溃。
- 显示设置中不再出现不可用的主题色占位。
- 图例、网格线、坐标轴使用 Switch 组件。
- 用户可以分别设置 X 轴和 Y 轴范围，也可以留空恢复自动范围。
- 旧工作台数据能继续以原始数据模式打开。

## 后续建议

- 第二阶段新增数据视图标签页，复用第一阶段聚合引擎。
- 第三阶段补充更多图表类型，如饼图、环形图、雷达图。
- 第四阶段扩展文件导入格式，如 Excel、JSON、TSV。
- 当多数据源字段不一致成为高频需求时，再设计字段映射能力。
