# CSV 可视化工作台 MVP 设计说明

## 目标

本项目将从“单图 CSV 可视化原型”重构为“通用 CSV 可视化工作台 MVP”。首版面向对外交付与演示场景，优先服务“懂数据但不懂前端/开发”的用户，让用户在上传结构规整的 CSV 后，立即得到可继续编辑的可视化结果。

MVP 的核心不是覆盖所有分析能力，而是建立一个可扩展的产品骨架：

- 一个数据集上下文
- 一个可自由增删的多图卡工作台
- 一套图卡内就地配置能力
- 一层通用但不过重的筛选、统计与性能降级机制

## 用户与场景

### 目标用户

- 懂数据、懂字段意义，但不想写脚本或前端代码的人
- 需要在本地或服务器静态部署后，向客户或团队直接演示 CSV 数据的人

### 核心场景

1. 用户打开网页，上传一个带表头且结构较规整的 CSV。
2. 系统自动生成一张默认折线图，避免空白首屏。
3. 用户继续添加散点图、柱状图或统计指标卡。
4. 用户在每张图卡内部调整字段、图表类型、坐标轴和样式。
5. 用户通过常用组合筛选缩小数据范围，并在多张图卡中观察同一份数据。

## 范围

### MVP 内

- 标准前端单页应用，支持本地运行与静态部署
- 上传单个 CSV 文件并解析
- 对空值、类型不符值按空处理
- 自动生成默认折线图
- 支持图卡类型：
  - 折线图
  - 散点图
  - 柱状图
  - 统计指标卡
- 多图卡工作台，可自由增删、复制、删除图卡
- 图卡内就地配置
- 全局组合筛选
- 大数据场景下自动抽样/降采样，优先保证交互流畅
- 记住界面配置，不恢复文件本体

### MVP 外

- 导出图片、导出配置文件
- 服务端解析、服务端聚合
- 脏数据智能修复
- 复杂查询语言
- 强联动的跨图刷选与同步缩放

## 产品结构

### 1. 数据入口层

负责文件上传、CSV 解析、字段归一化和数据集上下文建立。

规则：

- 仅支持带表头的 CSV
- 保留原始列顺序
- 每个单元格保留原始字符串，同时为数值型分析提供归一化结果
- 空值、无效数值统一视为 `null`

### 2. 工作台层

页面主体为多图卡工作区。所有图卡共享同一份经过筛选的数据集，但每张图卡有自己的配置与展示方式。

MVP 采用轻联动策略：

- 数据集共享
- 筛选共享
- 图卡配置独立

### 3. 图卡配置层

配置入口优先放在图卡内部，避免用户在全局侧栏和图表之间频繁切换。

每张图卡至少支持：

- 图卡标题编辑
- 图表类型切换
- X/Y 字段选择
- 样式基础调整
- 坐标轴基础调整
- 删除与复制

## 默认行为

上传成功后自动创建一张默认折线图。

默认规则：

- X 轴：第一列
- Y 轴：第二列
- 如果第二列无法用于数值绘图，则回退到“首个可用数值列且不等于第一列”
- 如果没有可绘制数值列，则不强行出图，改为提示用户添加统计卡或重新选列

## 数据与筛选模型

### 数据模型

每一行数据保留：

- 原始字段值
- 可用于数值计算的归一化值

这样可以兼顾：

- 文本筛选
- 数值绘图
- 对异常值按空处理

### 筛选能力

MVP 采用全局筛选条，支持常用组合筛选：

- 文本包含
- 文本等于
- 数值大于
- 数值小于
- 数值区间

筛选结果影响所有图卡，但图卡自身不再额外维护独立筛选状态。

## 图卡类型设计

### 折线图

- 适合顺序数据与趋势观察
- 默认首卡类型
- 大数据时进行自动降采样

### 散点图

- 适合观察两个字段之间的分布关系
- 使用同样的 X/Y 选择模型

### 柱状图

- MVP 先支持以行粒度绘制的简单柱状图
- 不做复杂聚合器

### 统计指标卡

统计信息本身被视为一种“图卡”。

MVP 指标卡展示：

- 非空数量
- 空值数量
- 最小值
- 最大值
- 均值
- 中位数

## 性能策略

产品目标是前端本地处理规整 CSV，并尽量支撑 10 万行以上场景。

MVP 采用“优先交互流畅”的策略：

- 图表渲染使用抽样/降采样
- 统计卡基于筛选后的完整数据计算
- 抽样阈值和目标点数固定在实现层，后续可调

## 持久化策略

MVP 仅持久化界面配置，不持久化文件内容。

持久化内容包括：

- 图卡列表
- 图卡配置
- 全局筛选条件

恢复原则：

- 新上传文件后尝试按列名兼容恢复
- 不兼容的字段配置自动回退为默认值

## 技术决策

### 保留 Vite + React + TypeScript

原因：当前项目已经具备前端骨架，重构成本低于另起炉灶。

### 去掉当前不稳定的 Tailwind/Vite 插件链路

原因：当前 `dev/build` 已被 Tailwind 原生依赖阻塞，而 MVP 重点在产品骨架，不在样式系统。

### 不使用额外第三方状态管理

原因：MVP 状态规模可由 React 本地状态和纯函数工具支撑，避免过早引入复杂度。

## 验收标准

1. `pnpm dev` 能正常启动页面。
2. 用户上传规整 CSV 后，页面自动生成默认折线图。
3. 用户可以新增至少 4 类图卡：折线图、散点图、柱状图、统计指标卡。
4. 用户可以在图卡内部修改字段与标题，并删除或复制图卡。
5. 用户可以配置多条全局筛选条件并看到所有图卡同步变化。
6. 遇到空值或非法数值时，系统不会报错中断，而是按空处理。
7. 对大数据集，图表可通过自动抽样保持基本可交互。
8. 刷新页面后，在重新上传兼容数据集时，可恢复上次界面配置。

## 风险与折中

### 大数据纯前端性能风险

折中方案：先用轻量抽样与纯函数处理保住可用性，不在 MVP 内追求完整分析引擎。

### 图卡内配置容易变复杂

折中方案：先只暴露常用配置项，把高级样式控制限制在少量字段中。

### 当前代码原型痕迹较重

折中方案：优先重建 App、数据模型和图卡体系，避免在旧单图逻辑上继续打补丁。
